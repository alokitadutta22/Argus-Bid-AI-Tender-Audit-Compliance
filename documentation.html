01Overview

Argus Bid AI is an evaluation platform for public-sector procurement. You give it the Master BID (the Notice Inviting Tender, or NIT) and the document sets submitted by each vendor; it returns a ranked, fully-justified compliance assessment you can defend in an audit.

The work splits into five stages: parse the tender, inventory each vendor's files, run the eligibility gates, score whoever survives, and explain the order. Every verdict carries the bid clause it rests on and the evidence text behind it, so nothing is a black box.

What it is not

Argus Bid AI does not replace the evaluation committee or make the award. It does the mechanical, error-prone reading and cross-checking at speed, and hands a transparent recommendation to the people who make the decision.

03Core concepts

03.1The BID / NIT parser

The parser reads the tender text line by line and extracts four things the rest of the pipeline depends on:

  • The Tender ID — the exact reference a vendor's Manufacturer's Authorization Form must cite.
  • Pre-Qualification Criteria (PQC) — eligibility thresholds such as minimum years of experience, minimum average annual turnover, and registration requirements.
  • Mandatory documents — the papers every vendor must submit (e.g. MAF, PAN, GST, GeM registration, audited balance sheet).
  • Technical specifications — split into mandatory specs (carry the 70% weight) and preferred specs (carry the remaining 30%).

03.2Inventory & readability

Vendors rarely name files helpfully. Argus Bid AI classifies each file by its content, not its name, so Scan_001.pdf is identified as a technical bid, a PAN card, or a GST certificate based on what it actually contains. Each file is then graded for legibility:

GradeMeaning
Passed OCRText extracted cleanly; the document is fully machine-readable.
Low-QualityText extracted, but with a high ratio of garbled characters — a scan worth a human glance.
CorruptedLittle to no usable text recovered; the document needs to be re-submitted or re-scanned.

03.3Eligibility gates

Before any score is calculated, each vendor passes through three gates. Failing any one of them disqualifies the vendor outright and the reason is written to an auditable log:

  • MAF gate — a Manufacturer's Authorization Form must be present, on OEM letterhead, and must cite the correct Tender ID. A missing form is Missing; a form that fails validation is Invalid.
  • PQC gate — the vendor must meet every pre-qualification threshold (experience, turnover, registration).
  • Mandatory-document gate — every required document must be present in the submission.

Disqualification vs deduction

Eligibility failures disqualify. A technical shortfall — a mandatory spec the vendor misses, or a value the vendor never stated — is not an automatic disqualification; it deducts points within the 70% bucket. This mirrors the brief's principle that every missed parameter costs points, while only missing documents and failed pre-qualification remove a bidder entirely.

03.4Technical comparison matrix

For every parameter in the tender, the matrix shows the BID requirement beside the vendor's stated value and a verdict chip:

  • meets / exceeds — the vendor satisfies the requirement.
  • below requirement — the vendor stated a value that falls short.
  • [DATA LACKING] — the vendor never stated the value, so it cannot be credited. This is treated as a miss, not assumed in the vendor's favour.

03.5Scoring model

Only vendors that clear all three gates are scored. The score is a transparent weighted sum:

# 70 points for mandatory specs, 30 for preferred
mandatory_part = 70 × (mandatory specs met ÷ total mandatory specs)
preferred_part = 30 × (preferred specs met ÷ total preferred specs)
score          = mandatory_part + preferred_part   # disqualified ⇒ 0

Worked example from the demo corpus: a vendor meeting all six mandatory specs earns the full 70; meeting two of four preferred features earns 15 of 30; total 85%. A vendor meeting everything scores 100%. The weighting is fixed and identical for every vendor, so the ranking is reproducible.

03.6Explainable ranking (XAI)

Argus Bid AI does not just print numbers — it explains the order in plain language. For adjacent ranks it names the specific parameters that separated them, for example which preferred feature the higher-ranked vendor satisfied that the runner-up did not. This is the "why did this vendor win" paragraph an evaluation committee needs on record.

04Architecture: rules decide, AI assists

This is the design choice that makes Argus Bid AI defensible, and it is worth understanding clearly.

The deterministic engine — always on

Every compliance verdict is produced by a rule-based engine. The same inputs always yield the same verdict, and each verdict carries the bid section and evidence behind it. That reproducibility is what survives an audit or a vendor challenge.

The AI layer — optional

An optional language-model layer does two things only: it re-classifies documents the rules could not confidently type, and it writes the executive narrative summary. It never overrides a verdict. If no key is supplied, or the call fails, the audit runs identically on rules alone — so a live demo cannot break.

In short: turning the AI off changes the prose, never the decision.

05Uploading documents

Supported formats are .pdf, .txt, and .md. For each vendor, give a name and attach all of their files together. For the cleanest extraction:

  • Prefer text-based PDFs over photographs of paper where possible.
  • Keep each vendor's documents under one vendor entry so the inventory and gates evaluate the full set.
  • If a file comes back graded Low-Quality, the text was still read but a human should verify it — the grade is a flag, not a rejection.

08FAQ

Does the AI make the final decision?

No. The rule engine makes every verdict. The AI only re-types ambiguous documents and writes the summary narrative, and can be switched off entirely without changing any result.

Why was a vendor with strong specs still disqualified?

Eligibility comes before merit. A missing mandatory document, a failed pre-qualification, or a missing/invalid MAF disqualifies regardless of how good the technical bid is. The disqualification log names the exact reason.

What does a 100% score mean?

The vendor met every mandatory spec (full 70) and every preferred feature (full 30). It is a relative compliance score against this tender, not an absolute quality rating.

Can I change the 70/30 weighting?

Yes — the split is defined in the scoring function in the engine. Keep it identical across all vendors in a given tender so the ranking stays reproducible and defensible.
